# 安全政策

## 支持范围

| 组件 | 说明 |
|------|------|
| f2b-sandbox 公网 API / 鉴权 | 优先 |
| f2b-web BFF（密钥边界） | 优先 |
| 官方 SDK（f2b-sdk-js / python） | 优先 |
| f2b-mcp-gateway / f2b-tunnel | 接受报告 |
| 依赖传递漏洞 | 请附可利用路径；仅版本号告警优先级较低 |
| 第三方部署 / 自改 fork | 自行负责；仍欢迎通用修复 PR |

**1.0 前**无长期 LTS 分支；修复默认合入各仓 `main`。

## 请报告的问题

- 未授权访问沙箱或跨租户数据  
- 管理密钥 / API Key 明文泄露到日志、前端 bundle、错误响应  
- 命令注入、路径穿越、SSRF（BFF 或服务代理）  
- 鉴权绕过（`F2B_AUTH_MODE=api_key` 下）  
- 供应链：恶意依赖、可疑 postinstall  

**不在此渠道**：功能请求、文档笔误、纯本地 Fake 行为疑问（请开 Issue）。

## 如何报告

请**不要**开公开 GitHub Issue 张贴可利用细节。

1. **首选**：GitHub **Private vulnerability reporting**（组织产品仓已于 2026-07-22 批量启用）：仓库 → Security → Report a vulnerability。  
2. 备选：邮件 **security@f2b.dev**（域名邮箱若尚未解析，可临时使用组织 Owner 联系邮箱，标题标明 `[SECURITY]`）。  

报告请尽量包含：

- 受影响仓与 commit / 版本  
- 复现步骤与环境（auth 模式、是否经 BFF）  
- 影响说明（读/写/执行边界）  
- 是否已有公开利用（避免扩大传播）

## 响应预期

| 阶段 | 目标（尽力） |
|------|----------------|
| 确认收到 | 3 个工作日内 |
| 初判严重度 | 7 个工作日内 |
| 修复或缓解方案 | 视严重度；Critical 优先 |

我们可能在修复就绪前要求暂缓完整公开细节（**协调披露**）。修复后可在 Release / 安全通告中致谢（除非你希望匿名）。

## 安全设计要点（贡献时自检）

1. **控制面 ≠ 数据面**：浏览器只走用户会话 / 用户 API Key → BFF 或公网 API；**永不**持有集群管理密钥。  
2. API Key **只存 hash**；创建响应中的明文 **只出现一次**。  
3. 用户输入进入 shell / 文件路径前必须约束。  
4. 服务端代理外部 URL 时防 SSRF。  
5. 日志、SSE、错误体不打印完整密钥。

## 内部流程演练

1.0 前至少完成一次**桌面演练**（虚构场景、不碰生产）：步骤见 [RELEASE.md §5](./RELEASE.md)。  
演练只验证渠道与响应模板，不替代真实漏洞修复。

## 致谢

感谢负责任的披露。目前无漏洞赏金计划；重大贡献可在安全通告中公开致谢。
